# 测试与截图验收策略

最后更新：2026-06-25

本文说明本项目如何验证行为正确性和视觉完成度。这里吸收了 `htdt/godogen` 中适合我们的部分：以截图和录屏为视觉事实来源，让捕获脚本保持确定性，并把视觉证据纳入完成标准。

## 行为回归测试

逻辑和交互路由使用 Godot headless 测试。

以下改动必须优先补测试：

- 用户已经反馈过的 bug。
- 输入路由变化。
- 任务进度。
- 背包、装备、掉落、属性。
- 战斗状态、伤害、胜利、失败、结算奖励。
- 可用节点可见性、文字、贴图路径、尺寸、位置断言的 UI 状态。

示例：

- 交互提示点击不能被地图点击寻路抢走。
- 统一面板快捷键应同时打开角色和背包。
- 从标题栏拖动面板后，面板位置应改变。
- 战斗结算奖励应进入 `PlayerData`。

## 视觉截图场景

主要是视觉改动时，必须使用 capture scene。

以下改动需要截图：

- HUD 布局。
- 战斗聚焦蒙版构图。
- 结算界面。
- 对话气泡位置。
- 上下文交互提示样式。
- 村庄或城堡地图构图。
- 主菜单、序章、词汇选择、背包、装备界面。

规则：

- Capture 脚本要显式设置待评审状态。
- 如果能直接触发目标状态，不要依赖脆弱的距离、时序或玩家手动操作。
- 截图保存到 `.tmp_assets/`。
- 截图是视觉评审依据，但不能替代行为测试。
- 目标设备要固定视口尺寸。单词城堡手机竖屏使用 720x1280。
- 声称视觉任务完成前，必须查看生成的截图。
- 代码判断和截图冲突时，以截图为准。

## 手动评审清单

交给用户 review 前，至少确认：

- 相关行为测试通过。
- 视觉改动有截图，或已明确进行人工截屏检查。
- 文字没有被裁切。
- 文字在容器内，并且留有舒适内边距。
- 按钮没有遮挡文字，也没有超出父面板。
- 主要 CTA 对比度清晰。
- 像素素材没有模糊。
- 精灵是完整物体，不是半个素材。
- 玩家能到达并离开交互区域。
- 玩家可见容器、按钮、边框、名牌使用已批准素材。
- 手机界面不显示桌面键盘提示。

## 视觉 Gate 命令

当前横屏村庄主线使用：

```bash
scripts/run_visual_gate.sh pc_village
```

该命令会执行：

- Godot 资源导入。
- 主菜单回归测试。
- 村庄视觉结构回归测试。
- 主菜单截图捕获。
- 村庄截图捕获。

输出截图：

```text
.tmp_assets/main_menu_visual_pass.png
.tmp_assets/main_menu_settings_visual_pass.png
.tmp_assets/main_menu_intro_visual_pass.png
.tmp_assets/village_visual_pass.png
```

单词城堡竖屏原型保留为长期参考：

```bash
scripts/run_visual_gate.sh word_castle_portrait
```

该命令会执行：

- Godot 资源导入。
- 单词城堡竖屏布局回归测试。
- 单词城堡竖屏截图捕获。

输出截图：

```text
.tmp_assets/word_castle_portrait_prototype.png
```
