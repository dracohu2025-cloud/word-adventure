# Godogen 工作流适配

来源参考：[htdt/godogen](https://github.com/htdt/godogen)

最后更新：2026-06-25

## 结论

`godogen` 对我们有价值的不是运行时代码，而是一套游戏制作闭环：先定义视觉目标，再实现可玩的垂直切片，之后用截图和视频证明结果真的符合设计。我们不引入它的 Godot C# 生成器，也不引入依赖外部大模型服务的资产流水线。

## 引入内容

### 1. 截图驱动修复

视觉改动不能只以“代码能跑”为完成标准。只要截图里出现以下问题，就视为未完成：

- 文字超出容器。
- 按钮遮挡文字。
- 容器边框尺寸不协调。
- 地图可视区域没有按目标设备铺满。
- 角色、怪物、NPC、道具比例不协调。
- 交互提示仍带桌面键盘文案。
- 玩家可见元素使用临时代码绘制，而不是素材。

### 2. 功能类型对应验证方式

- 纯逻辑改动：Godot headless 测试。
- UI 布局、地图、战斗表现、序章、主菜单：截图捕获测试。
- 动画和战斗节奏：短视频或连续帧，优先固定帧率。
- 新素材：先记录来源和许可，再运行 Godot import。

### 3. 任务记忆

长线程中不要依赖对话记忆。重要发现要沉淀到文件：

- `docs/production/project-memory.md`：项目级经验、坑、约定。
- `docs/production/playtest-notes.md`：用户体验反馈。
- `design/assets/asset-manifest.md`：素材来源、许可、用途。
- 对应规格文档：明确当前设计决策。

### 4. Godot API 不猜测

当 Godot API、节点行为、输入事件、Viewport 截图、布局容器、字体测量不确定时，先查官方文档或本地代码示例，不要凭印象写方法名或属性名。

## 不引入内容

- `godogen` 的 C#/.NET Godot 输出路径。
- 它的 `.claude` / `.agents` 生成器结构。
- 外部图像、视频、3D 服务依赖。
- 自动提交、自动推送、自动清理工作区。
- 与本项目 GDScript 架构冲突的 scene builder。

## 本项目执行方式

1. 读当前规格和相关场景脚本。
2. 先写或更新行为测试。
3. 对玩家可见改动写或更新 capture scene。
4. 运行测试和截图。
5. 查看截图，按视觉问题继续修。
6. 记录新发现。
7. 最终回复说明测试与截图路径。

当前入口：

```bash
scripts/run_visual_gate.sh word_castle_portrait
```
