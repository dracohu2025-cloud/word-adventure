# 单词城堡竖屏 UI 重设计方案

日期：2026-06-24

状态：长期方向参考，当前暂停；2026-06-25 起当前主线回到“横屏 + 书灯村 + PC Web 可玩性验证”

## 当前暂停说明

本设计记录保留“单词城堡”长期形态的思考，但不再作为当前迭代主线。

当前阶段优先级：

1. 横屏 `1280 x 720`。
2. 第一章“书灯村”。
3. PC Web 端可玩性验证。
4. 村庄探索、NPC 交互、词汇挑战、装备背包、实时战斗和结算闭环。

后续若重新启动“单词城堡”，应先从本文件和 `reference images/` 中恢复竖屏信息架构，但不能影响当前村庄版本推进。

## 背景

当前项目已经具备基础 RPG 系统：

- 主角移动、鼠标点击寻路和对象交互。
- NPC 对话、词汇挑战和剧情反馈。
- 实时数值战斗、伤害飘字和战斗结算。
- 玩家 HUD、背包、装备、物品品质和 tooltip。
- 序章、第一章、第二章的章节切换雏形。

但这些系统最初围绕横屏村庄和开放地图搭建。若回到最初设想的“单词城堡”，UI 必须重构为竖屏移动端优先，而不是继续压缩现有横屏 UI。

用户提供的 `reference images/` 截图显示了一个更适合本项目长期形态的方向：竖屏地牢房间、顶部状态栏、右上角小地图、底部快捷栏、全屏背包/技能面板，以及把单词学习嵌入门、机关、NPC 和战斗流程。

本设计只吸收参考图的信息架构和交互节奏，不复制其素材、图标、面板、按钮或具体视觉资产。

## 设计目标

- 将游戏主形态调整为 iPhone / iPad 竖屏优先。
- 支撑 50 层城堡、每层多个房间、每层约 20 个目标词。
- 支撑每层 20-30 个小怪、每 5 层一个 Boss。
- 支撑每层 NPC：交易、技能学习、词汇训练、治疗、楼层提示。
- 用移动端友好的方式重做 HUD、楼层地图、战斗、背包、装备、技能和 NPC 面板。
- 建立严格的 UI 防溢出规则，避免文字出框、按钮压字、面板拥挤。
- 继续贯彻视觉优先：所有玩家可见容器必须来自授权素材或生成素材。

## 已有经验必须迁移

“单词城堡”不是重新开一个没有历史包袱的新项目。之前在村庄、序章、战斗、背包和装备系统中暴露过的问题，必须作为新 UI 和新地图系统的硬约束迁移过来。

### UI 文字和容器

已经出现过的问题：

- 按钮文字上下没有居中。
- 左右两个按钮文字字号不一致。
- 继续按钮压住正文。
- 文本超过面板边框。
- 结算面板里的奖励文字贴近或越过底部边界。
- 选中态和未选中态视觉逻辑反了。
- 临时手绘边框观感差。

城堡版规则：

- 所有按钮文字必须水平、垂直居中。
- 同组按钮必须使用同字号、同高度、同内边距。
- 正文区域和按钮区域必须分离，不允许按钮覆盖正文。
- 所有正文 Label 必须开启自动换行，并保留安全内边距。
- 长文本超过可用高度时分页，不压缩字号硬塞。
- 选中态必须比未选中态更亮、更明确。
- 玩家可见面板、按钮、名牌、提示框必须使用素材或生成素材，不能用临时代码边框充当最终 UI。

### 地图道路和移动边界

已经出现过的问题：

- 道路太宽，导致路线引导弱。
- 道路太窄或碰撞体过硬，导致角色在边缘卡住。
- 角色看上去走到路外，破坏“只能沿路行动”的直觉。
- NPC、树、房屋、装饰物和路线重叠。
- 房屋、树木等素材曾经出现半截或拼装错误。

城堡版规则：

- 城堡房间不再依赖横向大路网，而用“房间 + 门 + 可行走区域”表达移动边界。
- 每个房间必须定义清晰的 `walkable_area`，角色只在可行走区域内移动。
- 可行走区域的视觉边缘和碰撞边缘必须留出角色半身宽的容错，不让角色视觉上踩出边界。
- 门口、NPC 前、宝箱前必须有稳定站位点，不能被装饰物或碰撞体挡住。
- 装饰物和交互物必须分层管理：装饰只负责氛围，不能侵占交互路径。
- 所有场景物体必须使用完整素材对象，不使用半截切片作为完整对象。

### 交互提示

已经出现过的问题：

- 叹号和交互按钮同时出现，信号冲突。
- 交互按钮颜色不明显。
- 提示容器尺寸不一致。
- 鼠标点击提示没有触发真实交互。

城堡版规则：

- 同一对象只保留一个主交互信号：优先使用高对比 CTA 按钮。
- 手机 / iPad 竖屏版默认只展示触控按钮，不展示 `空格/E`、方向键等桌面键盘提示。
- 键盘和手柄可以作为开发调试或桌面兼容输入，但不能进入移动端玩家可见 UI。
- 所有触控按钮、键盘操作和手柄操作必须调用同一底层方法。
- `交谈`、`挑战`、`打开`、`进入`、`学习` 等按钮必须尺寸一致、对齐一致。

### 战斗和结算

已经出现过的问题：

- 战斗双方距离太远，攻击感弱。
- 战斗面板太透明，焦点不集中。
- 大地图怪物血条造成视觉干扰。
- 战斗结算弹窗叠在战斗弹窗上。
- 结算文本和按钮挤出容器。

城堡版规则：

- 普通战斗在房间内就地发生，但必须有清晰受击反馈。
- Boss 战可以进入专门战斗构图，但双方距离必须足够近。
- 结算界面必须替代战斗界面，不允许弹窗套弹窗。
- 结算奖励列表必须有最大行数和行距；奖励过多时分页或合并，不允许越界。

### 素材一致性

已经出现过的问题：

- Kenney 和 Tiny Swords 在焦点区域混用时风格不一致。
- 某些装备图标类型不准确。
- 宝箱、道路、房屋、树木等对象需要完整视觉语义。

城堡版规则：

- 焦点对象优先使用 Tiny Swords 或 Tiny Swords 兼容生成素材。
- 其他开源素材只能作为风格审查通过后的补位。
- 每个可拾取物、装备、怪物、NPC 都要有清楚语义，不能用“看起来差不多”的素材临时代替。
- 新素材必须记录来源、授权和用途。

## 非目标

- 不 1:1 复刻任何参考游戏的视觉、素材或 UI 细节。
- 不在第一版实现全部 50 层。
- 不在第一版实现复杂随机迷宫生成。
- 不在第一版实现多职业、多语言包完整内容。
- 不继续维护横屏村庄 UI 作为主形态。

## 基础画布

首版按以下逻辑尺寸设计：

| 设备 | 逻辑尺寸 | 用途 |
| --- | --- | --- |
| iPhone 主目标 | `720 x 1280` | 所有 UI 的基准设计尺寸 |
| 小屏兜底 | `640 x 1136` | 检查文字和按钮是否仍可读 |
| iPad 竖屏 | `1024 x 1366` | 居中扩展，地图区域放大，UI 不无限拉伸 |

规则：

- 先按 `720 x 1280` 设计，再适配 iPad。
- 不以桌面横屏作为主设计尺寸。
- 竖屏主界面必须以地图视窗为主体：当前可视区域只是整层地图的一块局部裁切，顶部 HUD / 小地图和底部快捷栏浮在地图之上。
- UI 不能依赖 Godot 默认控件外观。

## 主界面结构

竖屏探索界面分为五层：

1. 顶部系统层。
2. 当前楼层信息层。
3. 地图可视层。
4. 上下文交互层。
5. 底部快捷操作层。

### 顶部系统层

顶部显示长期状态，不显示临时教学提示。

内容：

- 头像。
- 等级。
- HP / MP。
- 金币。
- 当前层数。
- 小地图入口。
- 暂停 / 设置入口。

建议布局：

```text
┌────────────────────────────┐
│ 头像 Lv. HP/MP      金币  小地图 │
└────────────────────────────┘
```

规则：

- HP / MP 数值与条形不得重叠。
- 金币必须使用金币图标，不使用木头、书页或临时代替图。
- 小地图入口固定在右上角，便于形成楼层认知。
- 顶部 HUD 最高占屏幕高度约 `12%-16%`。

### 当前楼层信息层

显示当前层目标，但不能常驻大段文字。

可显示：

- `第 1 层`
- `已掌握 3/20`
- `小怪 6/24`
- `Boss 未出现`

避免：

- 常驻任务说明长句。
- 长时间遮挡房间区域。
- 与顶部 HUD 重复的信息。

## 地图可视层

单词城堡不适合继续使用横向村庄大地图，也不应该把整层地图缩小塞进一个卡片里。推荐使用“整层地图 + 摄像机视窗”的表达：屏幕中看到的是地图局部，玩家移动时视窗跟随角色。

规则：

- 地图可视层应尽量铺满全屏。
- 顶部 HUD、小地图、底部物品栏是覆盖在地图上的 UI，不应把地图压缩成中间小窗口。
- 当前视窗外仍然存在房间、走廊、门和怪物；小地图负责表达整层结构。
- 对话、词汇选择和结算是临时浮层，只有触发时出现。

### 房间单位

每个房间是一屏内的可玩空间。

房间可包含：

- 门。
- 怪物。
- NPC。
- 宝箱。
- 机关。
- 楼梯。
- 词汇封印。
- 装饰物。

每个房间只能有少量交互焦点。第一版建议：

- 普通房间：1-2 个怪物。
- NPC 房间：1 个 NPC + 1 个功能对象。
- 商店房：1 个商人 + 货架/柜台。
- Boss 房：1 个 Boss + 入口封印。

### 门和房间切换

门是迷宫的核心交互。

规则：

- 房门必须有明确视觉状态：可进入、锁住、需要单词、Boss 门。
- 点击门时，若可进入，主角自动移动到门前并切换房间。
- 锁门不弹生硬系统错误，而是通过门牌、封印或 NPC 提示说明原因。

### 房间布局密度

竖屏房间不应堆满装饰。

建议比例：

- 可移动区域：`45%-60%`
- 固定场景装饰：`20%-35%`
- 交互对象：`5%-15%`
- UI 遮挡区：不超过 `20%`

## 楼层地图

每层需要一个可打开的大地图，右上角小地图只显示概览。

### 小地图

小地图固定在探索 HUD 右上角。

显示：

- 当前房间位置。
- 已探索房间。
- 未探索方向。
- NPC / 出口 / Boss 的简化标记。

不显示：

- 复杂文字。
- 每个怪物的精确位置。
- 大量图例。

### 大地图

大地图使用羊皮纸或石板风格全屏面板。

显示：

- 房间节点。
- 连接通道。
- 玩家当前位置。
- NPC 房。
- 商店房。
- 训练房。
- 存档点。
- Boss 房。
- 下一层出口。

地图图例必须短：

| 标记 | 含义 |
| --- | --- |
| 玩家头像 | 当前位置 |
| 门 | 可通行 |
| 锁 | 未解锁 |
| 书页 | 单词挑战 |
| 商人 | 商店 |
| 骷髅/怪物 | 战斗房 |
| 王冠 | Boss |

## 底部快捷操作层

底部是移动端最重要的操作区域。

固定入口：

- 背包。
- 技能。
- 词典 / 书页。
- 快捷物品槽。
- 当前主要交互按钮。

建议结构：

```text
┌────────────────────────────┐
│ 背包  技能  词典  药水  主要交互 │
└────────────────────────────┘
```

规则：

- 不再常驻“空格/E”这类桌面快捷提示。
- 移动端主按钮使用明确动词：`交谈`、`挑战`、`打开`、`进入`、`学习`。
- 如果保留键盘支持，只允许出现在桌面调试模式或设置页说明中，不占移动端主视觉。
- 当前没有可交互对象时，主交互按钮隐藏或灰显。

## 上下文交互

上下文交互必须靠近对象，但不能遮挡对象。

对象类型：

| 对象 | 按钮文案 |
| --- | --- |
| NPC | `交谈` |
| 怪物 | `挑战` |
| 宝箱 | `打开` |
| 门 | `进入` |
| 锁门 | `查看` |
| 技能导师 | `学习` |
| 商人 | `交易` |

规则：

- 鼠标点击、触摸点击和键盘交互必须调用同一底层方法。
- 不再同时显示叹号和上下文按钮。
- 上下文按钮必须有高对比 CTA 状态。
- 文案必须简短，优先两个汉字。

## 单词挑战 UI

单词学习不应像孤立考试，而应嵌入城堡机关。

推荐场景：

- 锁门：输入或选择正确单词解除封印。
- 书页：找回缺失词义。
- NPC：通过对话确认词义。
- 怪物：战斗前或战斗后触发轻量复习。
- 宝箱：答对后获得额外奖励。

### 选择题

布局：

```text
┌────────────────────────────┐
│ 这扇门在问你：“书”的英文是什么？ │
│                            │
│ [ book ]  [ fire ]  [ tree ] │
└────────────────────────────┘
```

规则：

- 中文问题使用自然口语，不使用“测试”“考察”等生硬词。
- 当前目标字/词使用强调色。
- 英文选项保持英文原文。
- 答对/答错反馈通过同一对话/机关流程继续，不额外塞一行系统提示。

### 拼写题

布局：

- 顶部：中文释义或语境句。
- 中部：输入槽。
- 下方：字母块。
- 右侧或底部：发音按钮。

规则：

- 输入框宽度必须容纳最长目标词。
- 字母块不得遮挡底部快捷栏。
- 音频按钮必须使用素材图标，不使用文字占位。

## 战斗 UI

城堡版战斗继续使用实时数值战斗，但竖屏重排。

### 普通战斗

普通怪物战斗应快速，不进入复杂全屏 Boss 视图。

建议：

- 房间内直接战斗。
- 怪物头顶显示血条。
- 角色和怪物靠近攻击。
- 底部技能栏可用。
- 伤害数字出现在受击者附近。

### Boss 战

Boss 战进入专门战斗构图。

布局：

```text
┌────────────────────────────┐
│ Boss 名称 + Boss HP         │
│                            │
│          Boss Avatar        │
│                            │
│      玩家 Avatar             │
│                            │
│ HP/MP + 技能按钮 + 药水       │
└────────────────────────────┘
```

规则：

- Boss 战必须比普通战斗更有仪式感。
- 玩家名称使用 `你`。
- 血条不得遮挡 avatar。
- 战斗结算必须替代战斗 UI，不允许弹窗叠弹窗。

## NPC UI

NPC 不只是剧情对话，还承担楼层功能。

### NPC 类型

| NPC | 功能 |
| --- | --- |
| 图书管理员 | 词汇讲解、词典解锁、楼层目标 |
| 商人 | 买卖消耗品、材料、低阶装备 |
| 技能导师 | 学习主动/被动技能 |
| 铁匠 | 装备强化或基础武器奖励 |
| 治疗师 | 回复 HP/MP |
| 楼层守卫 | Boss 门提示、楼层规则说明 |

### 对话面板

移动端对话面板推荐底部或中下部大面板。

内容：

- NPC 头像。
- NPC 名称。
- 对话正文。
- 1-3 个选项按钮。

规则：

- 不在对话框内混入独立系统反馈块。
- 选项按钮必须在正文下方，留足间距。
- 继续按钮不得覆盖文本。
- 长文本必须自动换行，并在超出高度前分页。

## 背包、装备、技能

城堡版不再使用横屏浮动窗。

### 全屏系统面板

背包、装备、技能、词典统一使用全屏纵向面板。

固定结构：

```text
┌────────────────────────────┐
│ 标题                         │
│ Tab / 分类                   │
│ 主内容区域                    │
│ 详情 / 操作区                 │
│ 返回 / 关闭                   │
└────────────────────────────┘
```

规则：

- 面板进入时遮罩探索场景。
- 面板不能互相叠放。
- 关闭按钮位置统一。
- Tab 选中态必须比未选中态更亮、更清晰。

### 背包

移动端背包首版建议：

- `5 x 4` 或 `4 x 5` 格。
- 每格大于 `72 x 72` 逻辑像素。
- 数量角标不贴边。
- 选中物品后在下方显示详情。

### 装备

装备面板建议：

- 上半：角色 avatar 和装备槽。
- 中部：装备属性。
- 下方：当前选中装备详情。

不再强行模仿桌面 WoW 左右多列装备窗。

### 技能

技能面板建议：

- 上半：技能点和分类。
- 中部：技能列表。
- 下方：选中技能详情和升级按钮。

技能需要区分：

- 主动技能。

## 视觉资产规则

所有 UI 容器必须来自以下来源之一：

1. Tiny Swords 已授权 UI 素材。
2. 已记录许可的开源 UI 素材。
3. 项目生成的像素 UI 素材，并记录来源和用途。

禁止：

- 直接用代码绘制临时边框作为正式 UI。
- 使用默认 Godot 按钮外观。
- 使用和 Tiny Swords 风格明显冲突的 UI 资产作为焦点容器。
- 复制参考截图里的素材。

## 文案规则

本项目面向中国大陆中小学生，中文必须自然。

规则：

- 不使用“冒险入口村”“新手入口村”这类功能性占位命名。
- 不使用 Chinglish 式直译。
- 村庄、楼层、房间和 NPC 都应有世界观内名称。
- UI 动词短而明确：`进入`、`交谈`、`挑战`、`打开`、`学习`。
- 解释性文字放在对话或词典中，不堆在主界面。
- 英文只在目标词、例句、拼写、物品英文名等学习场景出现。

命名建议：

| 概念 | 建议名称 |
| --- | --- |
| 单词城堡 | 万语之塔 / 词灵古堡 / 万语城 |
| 初始村庄 | 书灯村 |
| 第一层 | 旧书回廊 |
| 第二层 | 字母仓库 |
| 第五层 Boss 房 | 失名大厅 |

## 防溢出与 UI QA 规则

未来所有 UI 必须遵守“先布局、后填字、再截图验证”的流程。

### 文本容器规则

- 文本容器必须有明确 `custom_minimum_size`。
- 所有正文 Label 必须开启自动换行。
- 所有按钮文字必须居中。
- 按钮必须保留水平和垂直内边距。
- 任何按钮文字不得贴近边框。
- 长中文正文超过两行时必须分页，不强塞在一个面板。
- 选项按钮文案过长时优先缩短文案，不靠缩小字体硬塞。

### 逻辑尺寸断言

关键 UI 应补测试或调试断言：

- 文本矩形在父容器内。
- 继续按钮在面板内。
- 选项按钮不重叠。
- HUD 数值不压住血条。
- 背包格子数量和位置稳定。
- tooltip 不超出屏幕边界。

### 截图检查

每个视觉改动至少检查：

- `720 x 1280`
- `640 x 1136`
- `1024 x 1366`

截图检查目标：

- 无文字出框。
- 无按钮压字。
- 无 UI 互相遮挡。
- 主要 CTA 明显。
- 角色、怪物、NPC 不被 UI 长期遮住。

## 第一版原型范围

不直接重构全项目，先做一个可运行竖屏原型场景。

建议场景：

- `scenes/prototypes/word_castle_portrait_prototype.tscn`

首版原型包含：

- `720 x 1280` 竖屏画布。
- 顶部 HUD。
- 一个城堡房间。房间主体必须呈现城堡内部的房间迷宫，包括墙体、地砖、门和走廊；不得再用草地、房屋、树木等村庄户外元素占位。
- 一个门。
- 一个 NPC。
- 一个小怪。
- 一个宝箱。
- 底部快捷栏。
- 点击门切换到第二个房间。
- 打开小地图。
- 打开背包面板。
- 触发一个词汇封印问题。

这个原型的目标不是完整玩法，而是验证竖屏 UI 结构是否成立。

### 原型验收门槛

竖屏原型必须先通过以下检查，才允许迁移现有主线：

- 三个基准尺寸都有截图：`720 x 1280`、`640 x 1136`、`1024 x 1366`。
- 顶部 HUD、底部快捷栏、房间主体之间没有互相遮挡。
- 主要交互按钮在触摸尺寸上可用，按钮文字居中且不贴边。
- 至少一个对话、一个词汇挑战、一个宝箱、一个怪物交互可完整走通。
- 所有面板使用素材容器或生成容器，不出现临时手绘边框。
- 中文文案自然，不出现功能占位式命名。
- 没有任何文本超出容器或被按钮遮挡。

## 后续迁移策略

确认竖屏原型后，再按顺序迁移：

1. 主菜单进入“万语之塔”。
2. 序章结束进入城堡大厅，而不是横屏村庄。
3. 把当前新手村的三 NPC 支线拆成第一层三个功能房间。
4. 把 Word Imp Boss 改成第 1 层守门怪或第 5 层 Boss 的早期模板。
5. 把第二章森林改成第 2 层主题或城堡外侧支线。
